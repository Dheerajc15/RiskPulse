# RiskPulse — AI-Augmented Market Risk Intelligence Platform

A production-grade risk analytics system that combines quantitative finance (GARCH, VaR, Monte Carlo), distributed data processing (Dask), and AI-powered market intelligence (RAG over FOMC meeting minutes) through a FastAPI interface.

---

## What This System Does

RiskPulse ingests market data from Yahoo Finance and FRED, runs it through a cleaning and validation pipeline, computes institutional-grade risk metrics (VaR, CVaR, GARCH volatility, ADTV), and exposes everything through a REST API. A retrieval-augmented generation (RAG) pipeline adds natural-language market intelligence over a corpus of 42 FOMC meeting minutes (2021–2026) scraped directly from federalreserve.gov, with grounded answers generated by `gpt-4.1-mini`.

## Quick Start

### Prerequisites

- Python 3.10+
- pip
- An `OPENAI_API_KEY` in a `.env` file at the project root (required for `/rag/query`)

### Installation

```bash
git clone https://github.com/Dheerajc15/RiskPulse.git
cd RiskPulse
python -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### Build the FOMC corpus (one-time setup for RAG)

```bash
python scripts/build_corpus.py --years 5
```

This scrapes FOMC meeting minutes from federalreserve.gov for the last 5 years, validates each document, and writes them to `data/documents/fomc/` along with a manifest at `data/documents/corpus_manifest.json`. Expect ~40 documents and ~2.2M characters of cleaned text.

### Ingest into ChromaDB

```bash
python core/rag_chain.py
```

This chunks the corpus, embeds each chunk with `sentence-transformers/all-MiniLM-L6-v2`, and stores everything in a local ChromaDB at `data/chroma_db/`. Also runs three smoke-test queries.

### Run the API

```bash
uvicorn api.main:app --reload --port 8000
```

Open **http://localhost:8000/docs** for interactive Swagger UI.

### Run Tests

```bash
python tests/test_phase1.py            # OOP smoke tests
python tests/test_phase2_risk.py       # Risk engine tests
python tests/test_document_loader.py   # FOMC scraper unit tests (no network)
```

## Statistical Methodology

### GARCH (Generalized Autoregressive Conditional Heteroskedasticity)

GARCH models capture **volatility clustering** — the empirical observation that large price moves tend to follow large price moves. The GARCH(1,1) model defines conditional variance as:

```
σ²(t) = ω + α·ε²(t-1) + β·σ²(t-1)
```

- **ω (omega):** Long-run variance baseline
- **α (alpha):** Reaction to yesterday's shock. High α = volatility responds quickly to new information
- **β (beta):** Persistence of past volatility. High β = volatility decays slowly
- **Persistence (α + β):** Values near 1.0 indicate volatility shocks are long-lived. A persistence of 0.985 means a shock's half-life is ~46 trading days

RiskPulse uses the `arch` library with maximum-likelihood estimation under a Normal distribution assumption. See "Known Limitations" for why a Student-t distribution would be a future enhancement.

### Value at Risk (VaR)

VaR answers: "What is the maximum expected loss over one day at 95% confidence?"

**Historical VaR:** Sort all observed daily returns, take the 5th percentile. No distributional assumptions — purely empirical. Weakness: assumes the past distribution represents future risk.

**Monte Carlo VaR:** Simulate 10,000 return paths from a fitted Normal distribution (calibrated to observed μ and σ), then take the 5th percentile of simulated outcomes. Strength: extensible to multi-day horizons without overlapping windows. Weakness: Normal distribution underestimates tail risk for fat-tailed assets.

### CVaR (Conditional Value at Risk / Expected Shortfall)

CVaR answers: "If we DO breach the VaR threshold, how bad is the average loss?"

It is the mean of all returns worse than the VaR cutoff. Regulators (Basel III) prefer CVaR over VaR because VaR tells you the boundary of the worst 5%, while CVaR tells you what happens inside that tail.

### ADTV (Average Daily Trading Volume)

ADTV is the 20-day rolling mean of daily share volume. It is the primary liquidity metric used by institutional desks:

- **Why 20 days?** ≈ 1 trading month. Smooths out single-day spikes while remaining responsive
- **Liquidity flag:** If current ADTV falls below the 25th percentile of its own history, execution risk increases — larger orders will move the market

### Monte Carlo Simulation

The Monte Carlo engine simulates future cumulative returns by sampling from `N(μ, σ²)` calibrated to the observed log-return distribution. With 10,000 simulations over a configurable horizon, the 5th percentile of cumulative returns gives Monte Carlo VaR. The key advantage over historical VaR: you can extend the horizon (5-day, 10-day VaR) without needing overlapping historical windows.

## RAG Pipeline

### Corpus

42 FOMC meeting minutes spanning 2021-01-27 through 2026-03-18 (~2.2M characters total, mean ~53K characters per document). Built by `core/document_loader.py` via `scripts/build_corpus.py`:

1. Discover meeting dates by scraping the FOMC calendar pages
2. Fetch each minutes HTML with retry + polite 1-second delay
3. Strip nav/footer/scripts via BeautifulSoup; extract the article body
4. Validate against three quality gates: ≥5,000 chars, FOMC marker present, low boilerplate ratio
5. Write cleaned text to `data/documents/fomc/{date}_minutes.txt`
6. Write manifest with full provenance to `data/documents/corpus_manifest.json`

### Embedding & retrieval

- **Embedding model:** `sentence-transformers/all-MiniLM-L6-v2`
- **Vector store:** ChromaDB (persistent, local at `data/chroma_db/`)
- **Chunking:** 500 characters per chunk, 100-character overlap
- **Chunks indexed:** ~5,600

### Generation

- **Model:** `gpt-4.1-mini` (OpenAI), `temperature=0.2`, `max_tokens=500`
- **Prompt:** explicit grounding constraint that forces the model to acknowledge gaps rather than fabricate

### Evaluation

`notebooks/rag_market_data.ipynb` contains a five-section evaluation: corpus overview, retrieval quality across in-domain/adjacent/out-of-domain queries, chunk-size sensitivity (300/500/800 chars), model comparison (`gpt-4o-mini` vs `gpt-4.1-mini`), and prompt-engineering ablation showing how grounding constraints prevent hallucination.
